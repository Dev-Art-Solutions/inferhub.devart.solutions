InferHub

A self-hosted, Ollama-compatible inference mesh. Run the gateway where you have no GPU. Run worker nodes where you do.

InferHub puts a small, always-on coordinator in front of a pool of nodes that live on your GPU machines. The coordinator speaks an Ollama-shaped HTTP API; the nodes reach out to the coordinator over SignalR and pull jobs down — so they work fine behind NAT, with no port forwarding and nothing exposed to the public internet.

   

The node backend is powered by OllamaClient — our open-source .NET library for Ollama. The full library documentation, configuration, and API reference live at ollamaclient.devart.solutions.

The problem

GPUs and servers rarely live in the same place. The machine that is always on — a small VPS, a home server — usually has no GPU. The machines that do have a GPU — a desktop, a gaming rig, a workstation — are often behind a home router and not reachable from outside.

InferHub closes that gap. A lightweight coordinator runs on the always-on, GPU-less host and speaks a familiar Ollama-style API. The actual work runs on nodes that sit on your GPU machines, reach out to the coordinator, and pull jobs down.

In short: one stable address in front, a pool of GPUs behind it.

How it works

                                  ┌──────────────────────────────┐
   client (Ollama-compatible)     │          Coordinator         │
   curl / app / IDE plugin  ─────▶│  Ollama-style HTTP API       │
        (Bearer token if remote)  │  /api/tags /api/generate     │
                                  │  /api/chat                   │
                                  │                              │
                                  │  SignalR node hub  ◀─────────┼──── persistent outbound
                                  └───────────────┬──────────────┘     connection from nodes
                                                  │
                         dispatch (model-aware)   │   stream tokens back
                          ┌───────────────────────┼───────────────────────┐
                          ▼                       ▼                       ▼
                   ┌────────────┐          ┌────────────┐          ┌────────────┐
                   │   Node A   │          │   Node B   │          │   Node C   │
                   │  + Ollama  │          │  + Ollama  │          │  + Ollama  │
                   │  (GPU)     │          │  (GPU)     │          │  (GPU)     │
                   └────────────┘          └────────────┘          └────────────┘
  • Coordinator — an ASP.NET Core service. It exposes an Ollama-compatible HTTP API, authenticates remote callers with a Bearer token, keeps track of which nodes are online and which models each one holds, and routes every request to a capable node.
  • Node — a small .NET worker. It opens a persistent outbound connection to the coordinator (so it works fine behind NAT), reports which models it can serve, then receives prompts, runs them against its local inference backend, and streams the result back.
  • Pluggable backends — each node runs an LLM backend behind a small abstraction (IInferenceBackend). The first backend is Ollama, driven by our own OllamaClient library. Other backends (vLLM, llama.cpp, OpenAI-compatible servers) can slot in later without touching the coordinator.

Because the API is Ollama-shaped, existing Ollama clients, scripts, and editor plugins can point at the coordinator and keep working — they just reach a whole pool of GPUs instead of one.

Quick Start

One command with Docker, or two processes from source.

Docker

Fill in three keys, then bring the stack up:

cp deploy/docker/.env.example deploy/docker/.env   # set the three keys
docker compose -f deploy/docker/docker-compose.yml up -d

Images are published to GHCR for linux/amd64 and linux/arm64. See Docker for the full picture — including why the keys are mandatory there and optional below.

From source

On the always-on host (no GPU needed) start the coordinator:

dotnet run --project src/InferHub.Coordinator

On each GPU machine (with Ollama already running locally on :11434) start a node:

dotnet run --project src/InferHub.Node

From anywhere, talk to the coordinator like it is Ollama. Remote calls need a Bearer token:

curl http://your-coordinator:5080/api/chat \
  -H "Authorization: Bearer YOUR_API_KEY" \
  -d '{"model":"llama3","messages":[{"role":"user","content":"Hello!"}],"stream":false}'

That's it. The coordinator picks a node that holds llama3, runs the request there, and streams the response back. Prefer an OpenAI SDK? Point it at /v1 instead — see below.

OpenAI-compatible API

Almost everything in this ecosystem speaks the OpenAI wire format, and exposes exactly one knob for pointing somewhere new: a base URL.

Since v2.3.0 the coordinator serves an OpenAI-shaped surface at /v1, next to the Ollama endpoints it always had. Any OpenAI SDK, LangChain or LlamaIndex app, LiteLLM proxy, Continue/Cursor plugin, Open WebUI or n8n flow can talk to your own GPU mesh by changing one string.

EndpointNotes
POST /v1/chat/completionsBlocking and SSE streaming.
POST /v1/completionsLegacy text completion.
POST /v1/embeddingsfloat and base64 encodings — the Python SDK requests base64 by default.
GET /v1/models, GET /v1/models/{id}The models your nodes advertise.

The API key is an ordinary InferHub client key (Auth:ApiKeys) — the same one the Ollama surface uses:

from openai import OpenAI

client = OpenAI(base_url="http://your-coordinator:5080/v1", api_key="YOUR_API_KEY")

stream = client.chat.completions.create(
    model="llama3",
    messages=[{"role": "user", "content": "Explain NAT traversal in two sentences."}],
    stream=True,
)
for chunk in stream:
    print(chunk.choices[0].delta.content or "", end="")

Nothing about the mesh changed to make this work: the same least-busy router picks the node, the same sticky affinity keeps a conversation on the machine that already has the model warm, and the same pre-stream failover moves the job if that machine drops before the first token. The translation happens at the coordinator's front door — the nodes have no idea a second dialect exists.

Retrieval, for free

OpenAI clients let you set default headers, so a grounded answer over your own vector collection is a two-line change to an existing app rather than a new integration:

client = OpenAI(
    base_url="http://your-coordinator:5080/v1",
    api_key="YOUR_API_KEY",
    default_headers={"X-InferHub-Retrieve": "my-collection"},
)
Where the translation is lossy — stated, not papered over.
  • n > 1 is rejected with a 400, not quietly served once.
  • Tool calls are mapped in both blocking and streaming modes (since 2.9). A streamed call arrives as one delta.tool_calls frame with index: 0 and the terminal frame resolves finish_reason: tool_calls — Ollama emits the whole call at once, so we do not fake OpenAI-style argument-fragment streaming it never produced. The full loop (streamed call → tool result → grounded answer) works with the SDK's string-form arguments.
  • logprobs, logit_bias and user are accepted and ignored.
  • Image and other multimodal content parts are rejected, not silently dropped — a model should never answer confidently about an image it was never sent.

Errors on /v1/* use the OpenAI envelope ({"error": {"message", "type", "param", "code"}}), because an SDK reads error.message and would otherwise surface a useless "unknown error".

Authentication & configuration

InferHub keeps secrets out of source. Configure them at runtime via environment variables or .NET user-secrets — appsettings.json only ships empty placeholders.

Three independent token sets

Token setUsed byCoordinator config key
Client API keysInference callers — both dialects: /api/generate, /api/chat, /api/tags and everything under /v1Auth:ApiKeys
Admin API keysThe management console & /api/admin/* (cordon / drain / deregister)Auth:AdminApiKeys
Node enrollment secretWorker nodes joining the SignalR hubAuth:NodeEnrollmentSecret

Client and admin scopes are checked separately — an admin key cannot run inference unless it is also listed in Auth:ApiKeys, and vice versa. Since v2.7, entries in the flat Auth:ApiKeys list are anonymous clients with no limits — for named clients with quotas, see below.

Clients, quotas & usage (v2.7+)

A flat key list is fine until a second party is behind one of the keys. Auth:Clients gives a key an identity and, optionally, limits — every limit is nullable, and null means unlimited:

{
  "Auth": {
    "Clients": [
      {
        "Id": "acme-marketing",
        "Limits": {
          "MaxConcurrent": 4,
          "RequestsPerMinute": 60,
          "TokensPerMinute": 100000,
          "TokensPerDay": 2000000,
          "AllowedModels": ["llama3", "nomic-embed-text"]
        }
      }
    ]
  }
}

The key itself comes from env (Auth__Clients__0__Key=sk-...) or user-secrets — never a config file. A duplicate key across two clients fails startup, because attribution must never depend on list order.

What happens at the boundary, per status code:

SituationResponse
Over MaxConcurrent, RequestsPerMinute or TokensPerMinute429 with a window-accurate Retry-After
Over TokensPerDay402 Payment Required, Retry-After pointing at UTC midnight
Model outside AllowedModels404, byte-identical to a model that does not exist
Every capable node at its declared cap, longer than Queue:MaxWaitSeconds503 with Retry-After

On /v1 these use the OpenAI error envelope (rate_limit_exceeded, insufficient_quota), so SDK retry logic works out of the box.

Usage accounting. Every completed request is metered per client and per model — requests, prompt tokens, completion tokens, and whether it was served by cloud burst. Embeddings and document ingestion count too. Aggregates via GET /api/admin/usage?from&to&clientId&model, live window consumption via GET /api/admin/clients, and a Clients & usage console panel with CSV export.

What a usage record contains: a client id, a model name, a kind, two integers, a fallback flag and a timestamp. What it never contains: the prompt, the completion, a hash of either, or a "representative sample" — and there is no configuration flag to change that, because a flag is an invitation. Streaming counts come from the terminal chunk; a stream that never delivers it records nothing rather than guessing.

Counters are in-memory by default and a restart resets them. Set Usage:Persistence=postgres — with its own connection string, deliberately independent of the vector store's — to write each record to an append-only table.

Queueing. When every node holding a model is at its declared MaxConcurrency, a request waits in a bounded queue instead of failing instantly: up to Queue:MaxWaitSeconds (default 30), at most Queue:MaxDepth waiting (default 64), then 503. Nodes that declared no cap never queue. With cloud burst on no-node-or-saturated, saturation overflows to the upstream instead of queueing. Queue depth and median wait are on /api/status and the status page.

Loopback policy & production

By default, requests originating from 127.0.0.1 / ::1 skip the Bearer-token check — handy for local testing. Set Auth__RequireAuthForLoopback=true to require a token even for loopback. The same switch applies to both inference and admin routes. Remote (non-loopback) requests always require a valid token, regardless of this setting.

/health is unauthenticated so monitoring systems can poll it.

Production. Always run the coordinator behind HTTPS (a reverse proxy like Caddy / nginx, or Kestrel TLS). Bearer tokens are sensitive — don't send them over plain HTTP.

Inference backends

A node runs one backend behind the IInferenceBackend seam. The coordinator neither knows nor cares which one.

The coordinator hands a node an Ollama-shaped job and gets an Ollama-shaped response back, whatever actually ran it. Since v2.4 there are two implementations, selected by Backend:Type.

Backend:TypeDrivesWhy
ollama (default)OllamaOne machine, one model at a time, minimal ceremony.
openai (v2.4+)vLLMContinuous batching and paged attention — the difference between one GPU serving four users and one serving forty.
llama.cpp serverGGUF quants, CPU/Metal, a tiny footprint.
LM StudioA desktop model runner people already have open.
TGIHuggingFace's inference server.
Hosted providersOpenAI, or anything else wearing the same dialect.

One implementation covers all five, because they all converged on the same wire format. Three roadmap items — "vLLM backend", "llama.cpp backend", "hosted backend" — collapsed into one, entirely because v2.3's translation code could be run backwards.

// src/InferHub.Node/appsettings.json
{
  "Backend": { "Type": "openai" },
  "OpenAi": {
    "BaseUrl": "http://localhost:8000/v1",
    "TimeoutSeconds": 300,
    "Models": {
      "Include": ["meta-llama/Llama-3.1-8B-Instruct"]
    }
  }
}

Set OpenAi:ApiKey through the environment (OpenAi__ApiKey) or user-secrets — never in appsettings.json. A node configured with Backend:Type=openai and no BaseUrl refuses to start and names the missing key: a node that boots and then 500s on every job it is handed is worse than one that doesn't boot.

Against a hosted provider, an allowlist is effectively mandatory. A hosted catalogue is hundreds of models the node cannot actually serve. Report them all and the coordinator will happily route anything to it — so set OpenAi:Models:Include. vLLM and llama.cpp report only what they are serving, so there it is optional.

Digest and SizeBytes come back null: an OpenAI-compatible server reports a model name and nothing else, and inventing values would be worse than admitting it. /api/tags and the console render nulls as —.

Cloud burst (v2.4+)

When the router finds no node for a model, the coordinator can forward the request to a configured OpenAI-compatible upstream instead of returning 404. A GPU box that is switched off becomes degradation rather than an outage.

// src/InferHub.Coordinator/appsettings.json
{
  "Fallback": {
    "Enabled": true,
    "BaseUrl": "https://api.openai.com/v1",
    "Trigger": "no-node",                    // or "no-node-or-saturated"
    "ModelMap": { "llama3": "gpt-4o-mini" }, // the map is the consent
    "AllowedModels": []                      // empty = every mapped model
  }
}
This feature can send a user's prompt to a third party. Doing that by surprise, because somebody's desktop was asleep, is a betrayal rather than a feature. So it is fenced in, on purpose:
  • Off by default. Upgrading to v2.4 changes nothing.
  • Mapped models only. A model absent from ModelMap is never sent upstream, ever. There is no wildcard.
  • Always tagged. Every fallback response carries X-InferHub-Served-By: fallback; node-served responses carry node. Check the header, not your assumptions.
  • Always counted. /api/status and the status page report cloud burst and its counter whether it is on or off — "is this thing sending my prompts anywhere?" should not require reading a config file.
  • Never stored. The coordinator forwards in flight and streams straight through. It retains neither the prompt nor the answer — the same rule that has governed conversations since 0.7.

Trigger: no-node-or-saturated also bursts when every node holding the model is at its declared Node:MaxConcurrency. A node that declared no cap is never saturated — there is no number to compare against, and inventing one would burst to a paid upstream on a guess. And if a saturation burst fails while a node exists, the request goes to the node: bursting on saturation is an optimisation, not a promise.

With fallback disabled, a request for a model no node holds returns exactly the 404 it always has.

Running a node as a Windows service

Auto-start on boot, restart-on-failure, Event Viewer logging — one PowerShell install script.

For an always-on GPU box, run the node as a native Windows service instead of a console process. The service host (src/InferHub.Node.WindowsService) is a thin wrapper that composes the exact same node services through a shared AddInferHubNode root, so it behaves identically to dotnet run --project src/InferHub.Node — only the packaging differs. The dev/console and Linux paths are unchanged.

Publish a self-contained single-file host (no .NET runtime needed on the box):

dotnet publish src/InferHub.Node.WindowsService -c Release -r win-x64

Copy the output to C:\Program Files\InferHub\Node, set Coordinator:Url in appsettings.json, and set the enrollment secret as a machine environment variable (never commit it):

[Environment]::SetEnvironmentVariable('Coordinator__EnrollmentSecret','shared-node-secret','Machine')

Then install and start the service (run elevated):

./deploy/windows/install-service.ps1 `
  -BinaryPath "C:\Program Files\InferHub\Node\InferHub.Node.Service.exe" -DelayedStart

The install script configures automatic (or delayed-auto) start so the network and Ollama come up first, sets restart-on-failure recovery, and grants the service account write access to a writable data directory (Node:DataDirectory, default C:\ProgramData\InferHub\Node) so the node identity file survives under a least-privilege account. It logs to the Windows Event Log by default — check Event Viewer → Windows Logs → Application. The full runbook, including update/uninstall and virtual-account setup, lives in deploy/windows/README.md; a prebuilt win-x64 service + scripts are attached to the v2.1.0 release.

Linux? The equivalent is the same host pattern with builder.Services.AddSystemd() and a .service unit file — same composition root, different lifetime integration.

Docker

The coordinator is the piece that wants to be a container: stateless, always-on, and happiest on a small VPS.

cp deploy/docker/.env.example deploy/docker/.env   # set the three keys
docker compose -f deploy/docker/docker-compose.yml up -d

Published on every v* tag for linux/amd64 and linux/arm64, running as a non-root app user:

ghcr.io/dev-art-solutions/inferhub-coordinator:2.3.0   # also :2.3, :latest
ghcr.io/dev-art-solutions/inferhub-node:2.3.0
In Docker there is no loopback exemption — set the keys or nothing gets through.

On bare metal InferHub skips authentication for loopback callers, which is why the from-source quickstart lets you curl localhost with no key at all. That does not carry over: inside Docker your requests are not loopback — they arrive over the bridge network from outside the container — so every call needs a real Bearer token. This is the safer default and we left it alone, but it surprises people, so the compose stack refuses to start rather than quietly come up with an empty key list you cannot authenticate against.

The OpenAI base URL becomes http://localhost:5080/v1, the console is at /console, and /health stays open for monitoring.

Where the nodes should actually run

Management console & admin API

A browser console at /console (alias for /console.html) lets an operator drive the fleet — not just watch it. It is built from the same dark-theme HTML/CSS/JS as the status page (no build toolchain, no React) and uses the same admin endpoints any script can call.

What you can do

  • Cordon / Uncordon — flip a node out of (or back into) the routing pool. A cordoned node finishes its in-flight jobs and refuses new ones; the router silently skips it when picking candidates.
  • Drain — cordon, then wait for the node's local in-flight count to reach zero. The console implements this client-side as cordon + poll, so the request stays fast and the server never holds a long-lived connection.
  • Deregister — force-disconnect a node and drop it from the registry. If the worker process is still running it will reconnect cleanly and re-register.
  • Live updates — the console subscribes to GET /api/admin/stream (Server-Sent Events) and reflects node connect / disconnect / cordon changes in ~1s without a refresh. If the stream drops it transparently falls back to polling.
  • Last-action audit — each row shows the most recent admin action (who, when), kept in memory by the coordinator.
  • v1.9 — Vector collections panel. Per-collection record count, target vs live replicas with an under-replicated badge, holder node ids, and a Rebuild button that calls the admin rebuild endpoint.
  • v1.9 — Vector activity feed. Rolling live view of the last 40 vector events (created / dropped / assigned / lost / heal.started / heal.completed) with colour cues, over the same SSE stream.

Vector store

InferHub's vector store is provider-backed: VectorStore:Provider selects local (default) or, as of v2.2.0, postgres (external PostgreSQL + pgvector). Every endpoint, header, and client call is identical across providers — only the storage engine changes, and embeddings and inline retrieval always run on the fleet.

The local provider (since v1.5.0) is embedded on the coordinator. v1.6.0 closes the text loop. v1.7.0 spreads each collection's index across the connected nodes and lets queries be served by a node, not just the coordinator. v1.8.0 makes that placement self-healing — lose a holder and the index is rebuilt on a survivor from the raw store, automatically. v1.9.0 puts all of it on the console. v2.0.0 wires retrieval straight into /api/chat and /api/generate: the RAG mesh, GA. The local provider has two layers, kept honest:

  • Raw store — an append-only log of vector ops on disk (plus a periodic compacted snapshot) under VectorStore:DataDirectory. Plain newline-delimited JSON, one folder per collection. Source of truth, copyable, backupable. Lives on the coordinator.
  • Index — a flat (exact) cosine/dot/L2 structure in memory, rebuilt from the raw store at boot. Disposable. The coordinator always keeps a hub-local copy; from v1.7 it is also replicated to nodes.

It is off by default — flip VectorStore:Enabled to true to opt in. While disabled, the coordinator persists nothing, just like v1.4.

Replication across nodes (v1.7+)

With more than one node connected, each collection's index is replicated to a target number of them and queries can be answered by a node instead of only the coordinator. The split stays honest: the coordinator's raw store is the source of truth, and the node copies are derived from it — writes land in the raw store first, then fan out.

  • Placement. The coordinator targets VectorStore:ReplicationFactor replicas per collection (default 2, capped at the eligible node count). Cordoned nodes are skipped; placement is recomputed whenever the fleet changes (register, deregister, cordon, disconnect).
  • Push protocol over the existing SignalR link. Server → node messages (AssignVectorReplica seeds a snapshot, ApplyVectorOp forwards subsequent upserts/deletes, DropVectorReplica releases a holder). Nodes still expose nothing inbound.
  • Node-served queries. A new vector-query job kind threads through the dispatcher; reads prefer a node holding a replica (round-robin among the least-busy), falling back to the coordinator's hub-local index when none is available.
  • Hub-local index is the floor. Zero-node, one-node and many-node deployments all answer queries. The raw store on the always-on hub anchors the durability story.
  • Node-side persistence. Replicas are kept under Vector:ReplicaDirectory (default ./data/vector-replicas) using the same append-only-log + snapshot shape as the hub. The node rebuilds the index from disk at startup.
  • No full re-push on restart. The node reports its on-disk inventory (collection, lastSeq) when it registers; when the hub's lastSeq matches, the placement loop skips the re-push.

Self-healing (v1.8+)

v1.8 turns placement into a loop that keeps itself honest. A dedicated HealingService watches fleet changes (heartbeat eviction, deregister, cordon, disconnect) and the store's collection-created event; each event schedules a debounced heal pass that recomputes placement for every collection. A single in-flight gate makes the loop idempotent — a flapping fleet collapses into one settled pass rather than a storm.

  • Replica-loss detection. Subscribes to the existing lifecycle. When a holder drops, the collection falls below target and the loop re-pushes from the raw store to restore the factor.
  • Last-node-down recovery. When the live replica count for a collection hits zero, the hub-local index keeps serving reads (the floor since v1.7) and the next eligible node — a survivor, or the next to connect — is seeded from the raw store. No operator action needed.
  • New-node seeding. A node joining while the fleet is under target picks up a replica as part of the same heal loop.
  • Admin rebuild. POST /api/admin/vector/collections/{collection}/rebuild forces a heal-to-target re-push from the raw store; useful to proactively re-seed after a fleet event or to confirm replica integrity. Audit-logged as vector.rebuild.
  • Metrics. vectorReplicasHealed (each replica added by a heal), vectorRebuildsFromRaw (each last-node-down or admin rebuild), and vectorUnderReplicated (gauge of collections currently below target) join the metrics snapshot. They surface in the console in phase 17.

PostgreSQL + pgvector (v2.2+)

Set VectorStore:Provider to postgres and point the coordinator at a PostgreSQL that has the pgvector extension. You get HNSW-indexed ANN search, real transactions, ordinary database backups, and other apps reading the same table — with every endpoint, header, and client call unchanged. Pick it when you already run Postgres, want ANN + transactions + backups you understand, or the dataset has outgrown flat-exact search. Stay local when you want zero external services, plain-file backups, and search replicas living on the GPU fleet.

The IVectorStore seam was designed for exactly this since v1.5 — the postgres provider is the second implementation behind it, proving the swap is real rather than promised.

Schema

One registry table plus one table per collection (pgvector's vector(N) column is dimension-typed, so table-per-collection is the natural mapping). An ANN index is built per the collection's distance metric, and a GIN index backs metadata filters.

CREATE TABLE inferhub.collections (
    name text PRIMARY KEY, dimension integer NOT NULL,
    distance text NOT NULL, created_at timestamptz NOT NULL DEFAULT now());

CREATE TABLE inferhub.vec_docs (
    id text PRIMARY KEY, embedding vector(768) NOT NULL,
    payload jsonb, metadata jsonb,
    seq_no bigint NOT NULL, updated_at timestamptz NOT NULL DEFAULT now());
CREATE INDEX ON inferhub.vec_docs USING hnsw (embedding vector_cosine_ops)
    WITH (m = 16, ef_construction = 64);
CREATE INDEX ON inferhub.vec_docs USING gin (metadata);

Score sign-conventions match the local provider exactly, so clients see identical rankings and numbers: cosine → 1 - (embedding <=> $q) (higher is better), dot → -(embedding <#> $q) (higher is better), l2 → embedding <-> $q (lower is better). A cross-provider parity test pins identical ordering and scores within 1e-4 for all three metrics.

Honest trade-offs

Postgres is itself a durable, concurrently-queryable source of truth, so under this provider the coordinator holds no vector state on disk and does not replicate an index onto the fleet — two sources of truth would be a bug. Concretely:

  • No node replication, no self-healing, no node-served vector reads — search runs in Postgres; the replica/placement columns are zeroed and the vectorReplicasHealed / vectorRebuildsFromRaw / vectorUnderReplicated metrics stay flat.
  • The rebuild admin endpoint returns 409 (nothing to re-push), and the console disables its Rebuild button.
  • The status vector block carries a provider tag; the console shows a provider badge.
  • pgvector's ANN index tops out at 2000 dimensions — above that a collection still works but falls back to exact scan (logged at creation), never a silently broken index.
  • No migration path between providers yet — switching on a populated deployment means re-ingesting.

The mesh stays intact: embeddings and inline retrieval still run on the GPU nodes. Only the storage engine changed.

Document ingestion (v2.5+)

The vector store has existed since v1.5 and inline retrieval since v2.0 — but until v2.5 you had to fill the store yourself, with vectors you computed elsewhere or text pasted in by hand. Everything in between, the actual work of turning a folder of PDFs into something a model can answer from, was left as an exercise for the reader. v2.5 closes it: upload a document and the coordinator extracts its text, splits it into overlapping chunks, embeds those chunks on the GPU fleet — through the same dispatcher that serves chat, so the coordinator grows no embedding path of its own — and writes them to whichever vector store you configured.

# The collection must exist first: its dimension has to match your embedding model.
curl -X POST http://your-coordinator:5080/api/admin/vector/collections \

# Then upload. Text, Markdown, HTML, JSON, PDF.
curl -X POST http://your-coordinator:5080/api/collections/handbook/documents \

There is no OCR, and there never will be. A PDF whose text layer is empty or near-empty is rejected with an error saying it looks like a scan. It would have been easy to bolt on an OCR pass and produce something that usually works — but a bad extraction does not fail. It succeeds, quietly, and fills your corpus with near-gibberish that retrieves plausible nonsense; the failure surfaces months later as a model that is subtly, unaccountably wrong. We would rather refuse the file. If a document genuinely needs OCR, that is a decision its owner should make deliberately, with a tool they chose, before it reaches InferHub.

Three more decisions worth knowing about, because they are the ones that would have been easy to get wrong:

  • Your file is not kept. Chunk text, a content hash and metadata — not the document. A retrieval system that also quietly becomes a document store has two sources of truth and a data-retention question its owner never agreed to answer. If you need the original, you already have it.
  • Re-ingesting is idempotent. Chunk ids derive from the document id and the chunk index, so uploading a revision replaces its chunks rather than layering a second copy underneath the first — including sweeping the tail chunks when the revision is shorter, because a stale chunk retrieves as confidently as a live one. Upload identical bytes twice and the second call does no work and says "status": "unchanged".
  • A partial ingest is a failure and says so. If the fleet goes away mid-document, the response is 500 with "status": "partial" and the chunk counts. The chunks that landed are real and visible, the document lists as partial, and re-posting the same bytes resumes rather than no-ops. A half-ingested document that claims success is worse than a failure.

Ingestion is provider-agnostic: it works identically under the file-backed local store (where the new chunks replicate to your nodes through the usual path) and under PostgreSQL + pgvector. The embedding work is batched with a bounded number of chunks in flight, so ingesting a three-hundred-page manual does not starve the interactive chat traffic sharing the same GPUs. The console gained a Documents panel: pick a collection, drop a file, watch the chunk count climb, preview chunks, delete a document.

  -H "Authorization: Bearer YOUR_API_KEY" \

  -H "X-InferHub-Retrieve-K: 4" \
  -d '{"model":"llama3","messages":[{"role":"user","content":"What is InferHub?"}],"stream":false}'
  • Where the work happens. Only the search and the prompt assembly touch the hub. The embed job and the generation job are both dispatched to nodes over the existing SignalR link — no new machinery.
  • Query text. For chat, the last user message; for generate, prompt. If neither yields content, the request is passed through unchanged (or errors, per OnMissing).
  • Prompt assembly. Retrieved records are rendered as [id] text (drawn from payload.text, then payload.content, then the raw payload) and substituted into VectorStore:Retrieval:Template at the literal {context} placeholder. For chat the assembled context is inserted as a new system message after any operator-provided system messages; for generate it is prepended to prompt.
  • Failover. When retrieval can't run — unknown collection, no embedding node, empty query text, store disabled — VectorStore:Retrieval:OnMissing picks the behaviour: error returns 424 Failed Dependency; passthrough runs the original request unchanged and omits the sources header. Pre-stream failover on the generation job is unchanged.
  • Rule preserved: no conversation content on the hub. The augmented request body is assembled in-flight and forwarded to the node — nothing about the message or the retrieved context is retained on the coordinator.
  • Opt-in per request. A call without X-InferHub-Retrieve is byte-for-byte identical to v1.x — same routing, same streaming, same body.

Hybrid search & reranking (v2.6+)

Pure vector search is excellent at "what is this about" and poor at "find the exact thing I named" — ask for an error code, a part number or a surname and cosine similarity returns the general topic, not the line you wanted. v2.6 adds two answers, both per-request and both off by default.

ModeWhat it does
vectorDense embedding search — the pre-v2.6 behaviour, and the default.
keywordClassic BM25 full-text ranking over the same chunks. Provider-native: an in-memory inverted index under local, Postgres full-text (tsvector + GIN + ts_rank_cd) under postgres.
hybridRuns both branches and fuses them by Reciprocal Rank Fusion — by rank, not by blending scores that live on different scales. The one you usually want: it recovers the exact-match case without giving up the semantic one.
 
curl http://your-coordinator:5080/v1/chat/completions \
  -H "Authorization: Bearer YOUR_API_KEY" \
  -H "X-InferHub-Retrieve: handbook" \
  -H "X-InferHub-Retrieve-Mode: hybrid" \
  -d '{"model":"llama3","messages":[{"role":"user","content":"What does error E-4021 mean?"}]}'

Reranking (X-InferHub-Rerank: true) is an opt-in pass that hands the top candidates to a chat model already on your fleet with a scoring prompt and reorders them. It costs a round trip, so it is off unless asked, and it is hard-capped by Retrieval:RerankCandidates and Retrieval:RerankTimeoutSeconds — past the timeout the un-reranked order is kept. Nothing is retained. A dedicated cross-encoder fits behind the same seam later.

Defaults are unchanged. Retrieval:Mode defaults to vector and Retrieval:Rerank to none. A deployment that sends no headers and changes no config sees results identical to v2.5 — a feature that silently changes existing results is a regression wearing a feature's clothes.

Because "hybrid improves retrieval" is an empirical claim that depends on your corpus, v2.6 ships an evaluation harness in the same release — tools/InferHub.Eval runs a golden set against a live coordinator in every mode and reports Recall@k, MRR, nDCG@k and latency. Its README carries the one warning that matters: a golden set generated by the same model you are about to evaluate is a mirror, not evidence. Run it on your own data — that is the only number that is about you. The admin console also gains a query playground that shows what each mode retrieves for a query, side by side.

The RAG mesh in one paragraph

Six phases, one arc. v1.5 laid the raw store; v1.6 closed the text loop by moving embeddings to the fleet; v1.7 replicated collections across nodes; v1.8 made that placement heal itself; v1.9 put it all on the console; v2.0 wires retrieval straight into inference. That's a local vector store the operator can back up as plain files, replicated across the GPU fleet, that heals itself when nodes come and go, and that the models themselves can read from — with a single header and zero external services.

Resilience & failover

InferHub is built to keep running while nodes come and go.

  • Heartbeat eviction. Nodes that miss heartbeats past NodeRegistry:TimeoutSeconds are dropped from the registry; their in-flight jobs fail with a clear error and their sticky-conversation mappings are cleared.
  • Pre-stream failover. If the chosen node drops before the first chunk is produced, the coordinator transparently retries the request on another capable node (when one exists). This applies to both blocking and streaming calls.
  • No silent retry mid-stream. Once chunks have started flowing, the coordinator does not retry — the client already has partial output. Instead the stream ends with a final error chunk so callers don't hang.
  • Job timeout. Dispatcher:TimeoutSeconds caps how long any one job can hold a request open.

Fleet operations v2.8+

The coordinator can manage models across the fleet without an SSH session — over the same outbound connection the nodes already hold open, so a GPU node still needs no inbound port. Model commands travel down that channel exactly as inference jobs do.

Not every backend can manage models. An OpenAI-compatible node (vLLM, llama.cpp, a hosted provider) has its served model fixed at launch, so it declares that it cannot — the endpoints refuse it cleanly (never a 500) and the console greys out its controls. A seam that throws when you use it honestly is a seam nobody trusts twice.

Conversations & routing

InferHub stores no conversation content. Clients send the full message history on every turn, exactly like Ollama — the coordinator only decides which node runs the work.

  • Least-busy by default. When several nodes hold the requested model, the coordinator picks the one with the lowest in-flight job count and breaks ties round-robin.
  • Measured routing (v2.8, opt-in). Set Router:Strategy=throughput and the coordinator ranks capable nodes by expected completion time — a decayed average (EWMA) of measured tokens/second per model, adjusted for in-flight load — rather than raw queue depth. A 4090 and a laptop both reporting one job in flight stop being treated as equal. A node with no measurement is treated as average, never as slow, so a fresh node still earns traffic instead of being frozen out. The default stays least-busy and is unchanged bit-for-bit; measured tokens/sec appears on /api/status and the status page.
  • Sticky conversations. Successive turns of the same chat prefer the same node, which keeps that model's KV-cache warm. Affinity expires after ~10 idle minutes — and affinity wins over the throughput strategy, because a warm model on a slower node usually beats a cold one on a faster node.
  • Affinity guard. If the sticky node is far busier than the least-busy alternative, the coordinator breaks affinity to avoid piling up on one machine.
  • Graceful fallback. If the sticky node disconnects, the next turn transparently routes to another capable node and the mapping is refreshed.

Tagging a conversation. By default the coordinator hashes the opening system/user message to detect a continuing thread without any client cooperation. Clients that want explicit control can attach a stable id to every turn:

curl http://your-coordinator:5080/api/chat \
  -H "Authorization: Bearer YOUR_API_KEY" \
  -H "X-InferHub-Conversation: my-chat-7f3a" \
  -d '{"model":"llama3","messages":[...],"stream":false}'

The header value is opaque — any stable string identifying the conversation works.

C# client

InferHub.Client is a small, typed .NET client for the InferHub coordinator. Point it at a coordinator, pass a Bearer token, and call chat, generate, and the model list from C# with typed requests and dependency injection.

   

Targets net9.0 and net10.0. Serialization is System.Text.Json; the only runtime dependencies are Microsoft.Extensions.Http and the DI abstractions.

dotnet add package InferHub.Client

Wire it into a service collection and talk to the coordinator:

using InferHub.Client;
using InferHub.Client.Extensions;
using InferHub.Client.Models.Ollama;
using Microsoft.Extensions.DependencyInjection;

var services = new ServiceCollection();
services.AddInferHubClient(o =>
{
    o.BaseAddress = new Uri("http://localhost:5080");
    o.ApiKey = "sk-client-token-1";
});

var provider = services.BuildServiceProvider();
var client = provider.GetRequiredService<IInferHubClient>();

var chat = await client.ChatAsync(new ChatRequest
{
    Model = "llama3",
    Messages = new[]
    {
        new ChatMessage { Role = "user", Content = "Say hi in one word." }
    }
});

Console.WriteLine(chat.Message?.Content);

Streaming (v0.2.0). ChatStreamAsync and GenerateStreamAsync return IAsyncEnumerable<…> over the coordinator's NDJSON contract — one JSON object per line, stopping at the chunk with done:true. A terminal error chunk ({ "error": …, "done": true }) is surfaced as an InferHubException, so a partial answer plus a clean exception is the contract and the enumerator never hangs. Cancelling the token throws promptly out of the await foreach.

await foreach (var chunk in client.ChatStreamAsync(new ChatRequest
{
    Model = "llama3",
    Messages = new[] { new ChatMessage { Role = "user", Content = "Stream me a haiku." } }
}, cancellationToken))
{
    Console.Write(chunk.Message?.Content);

Embeddings (v0.3.0). EmbedAsync targets the modern batch endpoint POST /api/embed. Input is either a single string or a string[]; you get one vector per input back, in order. EmbedRequest.FromText(model, text) and EmbedRequest.FromTexts(model, texts) hide the JsonElement construction so the common cases are one line. EmbedLegacyAsync wraps the legacy POST /api/embeddings (single prompt) for drop-in Ollama callers. Missing embedding node → 404, bad body → 400, node dropped mid-flight → 502, all surfaced as InferHubException. An empty vector list on a 200 is treated as malformed and thrown too — you never get a silent zero-vector result.

var batch = await client.EmbedAsync(EmbedRequest.FromTexts(
    "nomic-embed-text",
    new[] { "InferHub", "self-hosted", "inference mesh" }));

// batch.Embeddings.Length == 3, one vector per input, same order.
Console.WriteLine(batch.Embeddings[0].Length); // model dimension

Vectors (v0.4.0). The vector data-plane lands: UpsertAsync, QueryAsync, RetrieveAsync, GetRecordAsync and DeleteRecordAsync, all under /api/vector/{collection}. Upsert a raw vector, or hand it text and a model and the coordinator embeds it on a node for you; query the same way — text in, ranked matches out, no embedding model held client-side. VectorUpsert.FromVector/FromText (plus WithPayload/WithMetadata) and VectorQuery.FromVector/FromText (plus WithFilter) keep the vector-or-text shapes discoverable. Typed VectorRecord/VectorMatch expose payload as a JsonElement? with a payload.As<T>() helper. GetRecordAsync returns null on a 404 and DeleteRecordAsync returns false; every other non-success status is an InferHubException. Needs the coordinator running with VectorStore:Enabled=true and the collection already created.

await client.UpsertAsync("docs", VectorUpsert
    .FromText("doc-1", "InferHub is a self-hosted inference mesh.", "nomic-embed-text"));

var matches = await client.QueryAsync("docs",
    VectorQuery.FromText("what is InferHub?", "nomic-embed-text", k: 3));

// matches[0].Score, matches[0].Id - ranked, closest first.

RAG retrieval (v0.5.0). Chat and generate can now be grounded in a collection per call. Pass an InferHubCallOptions — it maps to the coordinator's X-InferHub-Retrieve[-K|-Model] headers, so the request body stays untouched — and the coordinator pulls the top matches, grounds the prompt, and echoes the record ids it used in X-InferHub-Sources. The client parses those into SourceIds on the response (works for blocking and streaming). It ships as overloads, so a call with no options behaves exactly like a plain chat. A retrieval that cannot be satisfied — store unavailable, or OnMissing=error with no hits — fails with HTTP 424 as an InferHubRetrievalException (a subclass of InferHubException), never a silent, ungrounded answer. A per-call ConversationId maps to X-InferHub-Conversation for sticky routing.

var response = await client.ChatAsync(
    new ChatRequest
    {
        Model = "llama3",
        Messages = new[] { new ChatMessage { Role = "user", Content = "How do nodes connect?" } }
    },
    InferHubCallOptions.ForRetrieval("docs", k: 3, model: "nomic-embed-text"));

Console.WriteLine(response.Message?.Content);
Console.WriteLine(string.Join(", ", response.SourceIds ?? [])); // grounding record ids

Fleet + vector admin (v0.6.0). The admin planes land on a new IInferHubAdminClient — registered by the same AddInferHubClient(...) call but authenticated with AdminApiKey, and kept on its own interface so a client key alone never surfaces admin methods. Fleet ops: ListNodesAsync (in-flight counts, labels, cordon state, last audited action), CordonAsync/UncordonAsync/DeregisterAsync, plus a client-side DrainAsync extension that cordons a node and polls until its in-flight count hits zero. Vector admin: ListCollectionsAsync (with replica placement), GetCollectionAsync (placement, an underReplicated flag and query stats; null on 404), CreateCollectionAsync, DropCollectionAsync and RebuildAsync. StreamAdminEventsAsync tails the coordinator's SSE feed as an IAsyncEnumerable<AdminEvent> — fleet snapshots plus vector.* lifecycle events — and the AdminStreamOptions overload reconnects with exponential backoff (auth failures are never retried).

var admin = provider.GetRequiredService<IInferHubAdminClient>();

var drained = await admin.DrainAsync("node-1"); // cordon + wait for in-flight == 0
await admin.UncordonAsync("node-1");

await foreach (var ev in admin.StreamAdminEventsAsync(new AdminStreamOptions()))
{
    Console.WriteLine(ev.IsSnapshot
        ? $"fleet: {ev.Nodes!.Count} node(s)"
        : $"#{ev.Sequence} {ev.Event} {ev.Collection}");
}

Roadmap & status

InferHub 2.9 finishes tool calling on the OpenAI-compatible surface. Function calls were mapped in blocking mode only; now they stream too, as proper delta.tool_calls frames — so a streaming agent loop (the OpenAI SDK, LangChain, LlamaIndex) gets the call the moment it lands rather than a bare finish_reason with nothing attached. The whole call arrives in one delta and the terminal frame resolves finish_reason: tool_calls; we do not fabricate OpenAI-style argument-fragment streaming that Ollama never sent. Text-only streaming is byte-identical to before, the node protocol is unchanged, and the full call → tool-result → answer loop was verified live against a real GPU node. See OpenAI-compatible API.

That builds on 2.8, which makes a mixed fleet pleasant to run. The coordinator can finally act on the fleet's models — pull, delete and warm from the console or admin API, with a progress bar, over the connection each node already holds open (no inbound port), plus an ensure?replicas=N placement call and a fleet-wide model matrix. And the router stops pretending a 4090 and a laptop are the same node: opt-in Router:Strategy=throughput ranks nodes by measured tokens/second — an EWMA computed from data every job already reported — while an unmeasured node is treated as average, never starved, and sticky affinity still wins. The default routing is unchanged, bit for bit. See Fleet operations.

That builds on 2.7, which gives an API key a name, a budget, and a bill. Auth:Clients turns a flat list of anonymous keys into named clients with per-key concurrency, rate and token limits and a model allowlist; every completed request is metered per client and per model — counts, never text — queryable from the admin API, visible on the console, exportable as CSV, and optionally persisted to Postgres. Over a limit is a 429 with Retry-After; over the daily budget a 402; a fleet at capacity queues briefly, with a real bound, then says 503. Existing configs run unchanged. See Clients, quotas & usage.

That builds on 2.6, which goes after retrieval quality. Pure vector search is excellent at "what is this about" and poor at "find the exact thing I named", so retrieval now has three per-request modes — vector, keyword (BM25), and hybrid, which runs both and fuses them by rank (RRF) — plus an opt-in reranker that reuses a model already on your fleet. And because "hybrid improves retrieval" is a claim rather than a feeling, the same release ships an evaluation harness that measures it on your corpus. Defaults are unchanged: send no headers and results are identical to 2.5.

That builds on 2.5, which lets you fill the vector store at last. Upload a document — text, Markdown, HTML, JSON, or a PDF's text layer — and the coordinator extracts it, chunks it, embeds the chunks on your GPU fleet through the same dispatcher that serves chat, and stores them; any request carrying X-InferHub-Retrieve then comes back grounded, with citations down to the document and page. Three things it deliberately does not do: keep your file (chunks and a hash, nothing else), duplicate silently (re-ingesting replaces; identical bytes are a no-op and say so), or OCR (a scanned PDF is rejected with an error, not turned into a corpus of confident gibberish).

That builds on 2.4, which taught the mesh to speak the OpenAI format upstream, not just to its clients. A node is no longer synonymous with Ollama: set Backend:Type=openai and it drives vLLM — continuous batching, the difference between one GPU serving four users and one serving forty — or llama.cpp's server, LM Studio, TGI, or a hosted provider. One backend implementation covers all of them, because they all landed on the same dialect. That builds on 2.3, which made the mesh reachable from the tools you already use: an OpenAI-compatible API at /v1 alongside the Ollama endpoints, so any OpenAI SDK, LangChain app or IDE plugin can point at your own GPU pool by changing one base URL — with the same router, sticky affinity, failover and inline retrieval running underneath.
