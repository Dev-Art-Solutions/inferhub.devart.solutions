InferHub

A self-hosted, Ollama-compatible inference mesh. Run the gateway where you have no GPU. Run worker nodes where you do.

InferHub puts a small, always-on coordinator in front of a pool of nodes that live on your GPU machines. The coordinator speaks an Ollama-shaped HTTP API; the nodes reach out to the coordinator over SignalR and pull jobs down — so they work fine behind NAT, with no port forwarding and nothing exposed to the public internet.

   

The node backend is powered by OllamaClient — our open-source .NET library for Ollama. The full library documentation, configuration, and API reference live at ollamaclient.devart.solutions.

The problem

GPUs and servers rarely live in the same place. The machine that is always on — a small VPS, a home server — usually has no GPU. The machines that do have a GPU — a desktop, a gaming rig, a workstation — are often behind a home router and not reachable from outside.

InferHub closes that gap. A lightweight coordinator runs on the always-on, GPU-less host and speaks a familiar Ollama-style API. The actual work runs on nodes that sit on your GPU machines, reach out to the coordinator, and pull jobs down.

In short: one stable address in front, a pool of GPUs behind it.

How it works

                                  ┌──────────────────────────────┐
   client (Ollama-compatible)     │          Coordinator         │
   curl / app / IDE plugin  ─────▶│  Ollama-style HTTP API       │
        (Bearer token if remote)  │  /api/tags /api/generate     │
                                  │  /api/chat                   │
                                  │                              │
                                  │  SignalR node hub  ◀─────────┼──── persistent outbound
                                  └───────────────┬──────────────┘     connection from nodes
                                                  │
                         dispatch (model-aware)   │   stream tokens back
                          ┌───────────────────────┼───────────────────────┐
                          ▼                       ▼                       ▼
                   ┌────────────┐          ┌────────────┐          ┌────────────┐
                   │   Node A   │          │   Node B   │          │   Node C   │
                   │  + Ollama  │          │  + Ollama  │          │  + Ollama  │
                   │  (GPU)     │          │  (GPU)     │          │  (GPU)     │
                   └────────────┘          └────────────┘          └────────────┘
  • Coordinator — an ASP.NET Core service. It exposes an Ollama-compatible HTTP API, authenticates remote callers with a Bearer token, keeps track of which nodes are online and which models each one holds, and routes every request to a capable node.
  • Node — a small .NET worker. It opens a persistent outbound connection to the coordinator (so it works fine behind NAT), reports which models it can serve, then receives prompts, runs them against its local inference backend, and streams the result back.
  • Pluggable backends — each node runs an LLM backend behind a small abstraction (IInferenceBackend). The first backend is Ollama, driven by our own OllamaClient library. Other backends (vLLM, llama.cpp, OpenAI-compatible servers) can slot in later without touching the coordinator.

Because the API is Ollama-shaped, existing Ollama clients, scripts, and editor plugins can point at the coordinator and keep working — they just reach a whole pool of GPUs instead of one.

Quick Start

Two processes, then a single curl.

On the always-on host (no GPU needed) start the coordinator:

dotnet run --project src/InferHub.Coordinator

On each GPU machine (with Ollama already running locally on :11434) start a node:

dotnet run --project src/InferHub.Node

From anywhere, talk to the coordinator like it is Ollama. Remote calls need a Bearer token:

curl http://your-coordinator:5080/api/chat \
  -H "Authorization: Bearer YOUR_API_KEY" \
  -d '{"model":"llama3","messages":[{"role":"user","content":"Hello!"}],"stream":false}'

That's it. The coordinator picks a node that holds llama3, runs the request there, and streams the response back.

Authentication & configuration

InferHub keeps secrets out of source. Configure them at runtime via environment variables or .NET user-secrets — appsettings.json only ships empty placeholders.

Three independent token sets

Token setUsed byCoordinator config key
Client API keysInference callers (/api/generate, /api/chat, /api/tags)Auth:ApiKeys
Admin API keysThe management console & /api/admin/* (cordon / drain / deregister)Auth:AdminApiKeys
Node enrollment secretWorker nodes joining the SignalR hubAuth:NodeEnrollmentSecret

Client and admin scopes are checked separately — an admin key cannot run inference unless it is also listed in Auth:ApiKeys, and vice versa.

Setting secrets

Coordinator — client, admin & enrollment secrets:

# Linux / macOS
export Auth__ApiKeys__0="sk-client-token-1"
export Auth__ApiKeys__1="sk-client-token-2"
export Auth__AdminApiKeys__0="sk-admin-token"
export Auth__NodeEnrollmentSecret="shared-node-secret"

# Windows PowerShell
$env:Auth__ApiKeys__0 = "sk-client-token-1"
$env:Auth__AdminApiKeys__0 = "sk-admin-token"
$env:Auth__NodeEnrollmentSecret = "shared-node-secret"

Or with user-secrets (development):

dotnet user-secrets --project src/InferHub.Coordinator set "Auth:ApiKeys:0" "sk-client-token-1"
dotnet user-secrets --project src/InferHub.Coordinator set "Auth:AdminApiKeys:0" "sk-admin-token"
dotnet user-secrets --project src/InferHub.Coordinator set "Auth:NodeEnrollmentSecret" "shared-node-secret"

Node — enrollment secret:

export Coordinator__EnrollmentSecret="shared-node-secret"
# or
dotnet user-secrets --project src/InferHub.Node set "Coordinator:EnrollmentSecret" "shared-node-secret"

Loopback policy & production

By default, requests originating from 127.0.0.1 / ::1 skip the Bearer-token check — handy for local testing. Set Auth__RequireAuthForLoopback=true to require a token even for loopback. The same switch applies to both inference and admin routes. Remote (non-loopback) requests always require a valid token, regardless of this setting.

/health is unauthenticated so monitoring systems can poll it.

Production. Always run the coordinator behind HTTPS (a reverse proxy like Caddy / nginx, or Kestrel TLS). Bearer tokens are sensitive — don't send them over plain HTTP.

Management console & admin API

A browser console at /console (alias for /console.html) lets an operator drive the fleet — not just watch it. It is built from the same dark-theme HTML/CSS/JS as the status page (no build toolchain, no React) and uses the same admin endpoints any script can call.

What you can do

  • Cordon / Uncordon — flip a node out of (or back into) the routing pool. A cordoned node finishes its in-flight jobs and refuses new ones; the router silently skips it when picking candidates.
  • Drain — cordon, then wait for the node's local in-flight count to reach zero. The console implements this client-side as cordon + poll, so the request stays fast and the server never holds a long-lived connection.
  • Deregister — force-disconnect a node and drop it from the registry. If the worker process is still running it will reconnect cleanly and re-register.
  • Live updates — the console subscribes to GET /api/admin/stream (Server-Sent Events) and reflects node connect / disconnect / cordon changes in ~1s without a refresh. If the stream drops it transparently falls back to polling.
  • Last-action audit — each row shows the most recent admin action (who, when), kept in memory by the coordinator.
  • v1.9 — Vector collections panel. Per-collection record count, target vs live replicas with an under-replicated badge, holder node ids, and a Rebuild button that calls the admin rebuild endpoint.
  • v1.9 — Vector activity feed. Rolling live view of the last 40 vector events (created / dropped / assigned / lost / heal.started / heal.completed) with colour cues, over the same SSE stream.

Vector store

As of v1.5.0 InferHub ships with a local, embedded vector store on the coordinator. v1.6.0 closes the text loop. v1.7.0 spreads each collection's index across the connected nodes and lets queries be served by a node, not just the coordinator. v1.8.0 makes that placement self-healing — lose a holder and the index is rebuilt on a survivor from the raw store, automatically. v1.9.0 puts all of it on the console. v2.0.0 wires retrieval straight into /api/chat and /api/generate: the RAG mesh, GA.

Two layers, kept honest:

  • Raw store — an append-only log of vector ops on disk (plus a periodic compacted snapshot) under VectorStore:DataDirectory. Plain newline-delimited JSON, one folder per collection. Source of truth, copyable, backupable. Lives on the coordinator.
  • Index — a flat (exact) cosine/dot/L2 structure in memory, rebuilt from the raw store at boot. Disposable. The coordinator always keeps a hub-local copy; from v1.7 it is also replicated to nodes.

It is off by default — flip VectorStore:Enabled to true to opt in. While disabled, the coordinator persists nothing, just like v1.4.

Text in, vectors out (v1.6+)

From v1.6 onwards upsert and query accept text in place of a raw vector. The coordinator dispatches an embed job to a node advertising the requested embedding model, then stores or searches with the returned vector. The raw-vector path from v1.5 is unchanged.

curl -X POST http://your-coordinator:5080/api/vector/docs/upsert \
  -H "Authorization: Bearer YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{"id":"doc-1","text":"InferHub now embeds on the fleet.","model":"nomic-embed-text"}'
curl -X POST http://your-coordinator:5080/api/vector/docs/retrieve \
  -H "Authorization: Bearer YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{"text":"how does embedding work?","k":4}'

VectorStore:DefaultEmbeddingModel kicks in when model is omitted. The collection's dimension is validated against the returned embedding length — a mismatch surfaces as a clear 400; no embedding node available surfaces as a 404 naming the model.

Replication across nodes (v1.7+)

With more than one node connected, each collection's index is replicated to a target number of them and queries can be answered by a node instead of only the coordinator. The split stays honest: the coordinator's raw store is the source of truth, and the node copies are derived from it — writes land in the raw store first, then fan out.

  • Placement. The coordinator targets VectorStore:ReplicationFactor replicas per collection (default 2, capped at the eligible node count). Cordoned nodes are skipped; placement is recomputed whenever the fleet changes (register, deregister, cordon, disconnect).
  • Push protocol over the existing SignalR link. Server → node messages (AssignVectorReplica seeds a snapshot, ApplyVectorOp forwards subsequent upserts/deletes, DropVectorReplica releases a holder). Nodes still expose nothing inbound.
  • Node-served queries. A new vector-query job kind threads through the dispatcher; reads prefer a node holding a replica (round-robin among the least-busy), falling back to the coordinator's hub-local index when none is available.
  • Hub-local index is the floor. Zero-node, one-node and many-node deployments all answer queries. The raw store on the always-on hub anchors the durability story.
  • Node-side persistence. Replicas are kept under Vector:ReplicaDirectory (default ./data/vector-replicas) using the same append-only-log + snapshot shape as the hub. The node rebuilds the index from disk at startup.
  • No full re-push on restart. The node reports its on-disk inventory (collection, lastSeq) when it registers; when the hub's lastSeq matches, the placement loop skips the re-push.

Self-healing (v1.8+)

v1.8 turns placement into a loop that keeps itself honest. A dedicated HealingService watches fleet changes (heartbeat eviction, deregister, cordon, disconnect) and the store's collection-created event; each event schedules a debounced heal pass that recomputes placement for every collection. A single in-flight gate makes the loop idempotent — a flapping fleet collapses into one settled pass rather than a storm.

  • Replica-loss detection. Subscribes to the existing lifecycle. When a holder drops, the collection falls below target and the loop re-pushes from the raw store to restore the factor.
  • Last-node-down recovery. When the live replica count for a collection hits zero, the hub-local index keeps serving reads (the floor since v1.7) and the next eligible node — a survivor, or the next to connect — is seeded from the raw store. No operator action needed.
  • New-node seeding. A node joining while the fleet is under target picks up a replica as part of the same heal loop.
  • Admin rebuild. POST /api/admin/vector/collections/{collection}/rebuild forces a heal-to-target re-push from the raw store; useful to proactively re-seed after a fleet event or to confirm replica integrity. Audit-logged as vector.rebuild.
  • Metrics. vectorReplicasHealed (each replica added by a heal), vectorRebuildsFromRaw (each last-node-down or admin rebuild), and vectorUnderReplicated (gauge of collections currently below target) join the metrics snapshot. They surface in the console in phase 17.

Console & observability (v1.9+)

v1.9 makes the mesh visible and operable. Nothing on the write path changed — this is pure surface area.

  • Status vector block. GET /api/status now returns per-collection record count, dimension, distance, target vs live replicas, holder node ids, and an underReplicated flag. Omitted when the store is disabled, so consumers that ignored the field before see no change.
  • Live vector events on /api/admin/stream. Six new event kinds ride the same SSE channel as the fleet snapshot: vector.collection.created, vector.collection.dropped, vector.replica.assigned, vector.replica.lost, vector.heal.started, vector.heal.completed. Each carries a monotonic sequence, an atUtc timestamp, the collection, and a data blob (holder connection id, node id, reason, before/after counts).
  • Console — Collections panel. Record count, replication factor (target vs current), replica placement per node, and a badge that flips to "under-replicated" when a collection drops below target. Each row has a Rebuild button that calls the admin rebuild endpoint.
  • Console — Vector activity feed. Rolling live feed of the last 40 vector events (newest first) with per-kind colour cues. Updates over SSE with the polling fallback the console already had for node snapshots.
  • Admin detail endpoint. GET /api/admin/vector/collections/{collection} returns collection info, placement, the underReplicated flag, and per-collection query stats (queries, queryLatencyAvgMs).
  • Per-collection query stats. The metrics snapshot gains a perCollection array — query count and average latency in milliseconds — recorded on every /query and /retrieve call, regardless of whether the query was served hub-local or by a node replica.
  • Build-free. Same dark-theme HTML/CSS/JS as v1.3, no framework, no toolchain. The vector event bus uses bounded queues with drop-oldest semantics so a slow subscriber can never back-pressure the write path.

Retrieval-augmented inference (v2.0+)

v2.0 closes the RAG mesh: models can read from the store inline. A client opts a normal /api/chat or /api/generate call into retrieval by naming a collection; the coordinator embeds the query on a node, searches the store, injects the top matches as context, and dispatches the generation to a node. The Ollama-shaped body stays pure — retrieval is driven entirely by headers, and the retrieved source ids come back on a response header.

HeaderDirectionPurpose
X-InferHub-RetrieverequestCollection name. Presence enables RAG for this call.
X-InferHub-Retrieve-KrequestTop-k override, clamped to VectorStore:Retrieval:MaxRecords.
X-InferHub-Retrieve-ModelrequestEmbedding model override (defaults to VectorStore:DefaultEmbeddingModel).
X-InferHub-SourcesresponseJSON array of the retrieved record ids used to ground the answer.

Example — a grounded chat call:

curl http://your-coordinator:5080/api/chat \
  -H "Authorization: Bearer YOUR_API_KEY" \
  -H "X-InferHub-Retrieve: docs" \
  -H "X-InferHub-Retrieve-K: 4" \
  -d '{"model":"llama3","messages":[{"role":"user","content":"What is InferHub?"}],"stream":false}'
  • Where the work happens. Only the search and the prompt assembly touch the hub. The embed job and the generation job are both dispatched to nodes over the existing SignalR link — no new machinery.
  • Query text. For chat, the last user message; for generate, prompt. If neither yields content, the request is passed through unchanged (or errors, per OnMissing).
  • Prompt assembly. Retrieved records are rendered as [id] text (drawn from payload.text, then payload.content, then the raw payload) and substituted into VectorStore:Retrieval:Template at the literal {context} placeholder. For chat the assembled context is inserted as a new system message after any operator-provided system messages; for generate it is prepended to prompt.
  • Failover. When retrieval can't run — unknown collection, no embedding node, empty query text, store disabled — VectorStore:Retrieval:OnMissing picks the behaviour: error returns 424 Failed Dependency; passthrough runs the original request unchanged and omits the sources header. Pre-stream failover on the generation job is unchanged.
  • Rule preserved: no conversation content on the hub. The augmented request body is assembled in-flight and forwarded to the node — nothing about the message or the retrieved context is retained on the coordinator.
  • Opt-in per request. A call without X-InferHub-Retrieve is byte-for-byte identical to v1.x — same routing, same streaming, same body.

The RAG mesh in one paragraph

Six phases, one arc. v1.5 laid the raw store; v1.6 closed the text loop by moving embeddings to the fleet; v1.7 replicated collections across nodes; v1.8 made that placement heal itself; v1.9 put it all on the console; v2.0 wires retrieval straight into inference. That's a local vector store the operator can back up as plain files, replicated across the GPU fleet, that heals itself when nodes come and go, and that the models themselves can read from — with a single header and zero external services.

Resilience & failover

InferHub is built to keep running while nodes come and go.

  • Heartbeat eviction. Nodes that miss heartbeats past NodeRegistry:TimeoutSeconds are dropped from the registry; their in-flight jobs fail with a clear error and their sticky-conversation mappings are cleared.
  • Pre-stream failover. If the chosen node drops before the first chunk is produced, the coordinator transparently retries the request on another capable node (when one exists). This applies to both blocking and streaming calls.
  • No silent retry mid-stream. Once chunks have started flowing, the coordinator does not retry — the client already has partial output. Instead the stream ends with a final error chunk so callers don't hang.
  • Job timeout. Dispatcher:TimeoutSeconds caps how long any one job can hold a request open.

Conversations & routing

InferHub stores no conversation content. Clients send the full message history on every turn, exactly like Ollama — the coordinator only decides which node runs the work.

  • Least-busy by default. When several nodes hold the requested model, the coordinator picks the one with the lowest in-flight job count and breaks ties round-robin.
  • Sticky conversations. Successive turns of the same chat prefer the same node, which keeps that model's KV-cache warm. Affinity expires after ~10 idle minutes.
  • Affinity guard. If the sticky node is far busier than the least-busy alternative, the coordinator breaks affinity to avoid piling up on one machine.
  • Graceful fallback. If the sticky node disconnects, the next turn transparently routes to another capable node and the mapping is refreshed.

Tagging a conversation. By default the coordinator hashes the opening system/user message to detect a continuing thread without any client cooperation. Clients that want explicit control can attach a stable id to every turn:

curl http://your-coordinator:5080/api/chat \
  -H "Authorization: Bearer YOUR_API_KEY" \
  -H "X-InferHub-Conversation: my-chat-7f3a" \
  -d '{"model":"llama3","messages":[...],"stream":false}'

The header value is opaque — any stable string identifying the conversation works.

C# client

InferHub.Client is a small, typed .NET client for the InferHub coordinator. Point it at a coordinator, pass a Bearer token, and call chat, generate, and the model list from C# with typed requests and dependency injection.

   

Targets net9.0 and net10.0. Serialization is System.Text.Json; the only runtime dependencies are Microsoft.Extensions.Http and the DI abstractions.

dotnet add package InferHub.Client

Wire it into a service collection and talk to the coordinator:

using InferHub.Client;
using InferHub.Client.Extensions;
using InferHub.Client.Models.Ollama;
using Microsoft.Extensions.DependencyInjection;

var services = new ServiceCollection();
services.AddInferHubClient(o =>
{
    o.BaseAddress = new Uri("http://localhost:5080");
    o.ApiKey = "sk-client-token-1";
});

var provider = services.BuildServiceProvider();
var client = provider.GetRequiredService<IInferHubClient>();

var chat = await client.ChatAsync(new ChatRequest
{
    Model = "llama3",
    Messages = new[]
    {
        new ChatMessage { Role = "user", Content = "Say hi in one word." }
    }
});

Console.WriteLine(chat.Message?.Content);

Streaming (v0.2.0). ChatStreamAsync and GenerateStreamAsync return IAsyncEnumerable<…> over the coordinator's NDJSON contract — one JSON object per line, stopping at the chunk with done:true. A terminal error chunk ({ "error": …, "done": true }) is surfaced as an InferHubException, so a partial answer plus a clean exception is the contract and the enumerator never hangs. Cancelling the token throws promptly out of the await foreach.

await foreach (var chunk in client.ChatStreamAsync(new ChatRequest
{
    Model = "llama3",
    Messages = new[] { new ChatMessage { Role = "user", Content = "Stream me a haiku." } }
}, cancellationToken))
{
    Console.Write(chunk.Message?.Content);
}

Embeddings (v0.3.0). EmbedAsync targets the modern batch endpoint POST /api/embed. Input is either a single string or a string[]; you get one vector per input back, in order. EmbedRequest.FromText(model, text) and EmbedRequest.FromTexts(model, texts) hide the JsonElement construction so the common cases are one line. EmbedLegacyAsync wraps the legacy POST /api/embeddings (single prompt) for drop-in Ollama callers. Missing embedding node → 404, bad body → 400, node dropped mid-flight → 502, all surfaced as InferHubException. An empty vector list on a 200 is treated as malformed and thrown too — you never get a silent zero-vector result.

var batch = await client.EmbedAsync(EmbedRequest.FromTexts(
    "nomic-embed-text",
    new[] { "InferHub", "self-hosted", "inference mesh" }));

// batch.Embeddings.Length == 3, one vector per input, same order.
Console.WriteLine(batch.Embeddings[0].Length); // model dimension

Vectors (v0.4.0). The vector data-plane lands: UpsertAsync, QueryAsync, RetrieveAsync, GetRecordAsync and DeleteRecordAsync, all under /api/vector/{collection}. Upsert a raw vector, or hand it text and a model and the coordinator embeds it on a node for you; query the same way — text in, ranked matches out, no embedding model held client-side. VectorUpsert.FromVector/FromText (plus WithPayload/WithMetadata) and VectorQuery.FromVector/FromText (plus WithFilter) keep the vector-or-text shapes discoverable. Typed VectorRecord/VectorMatch expose payload as a JsonElement? with a payload.As<T>() helper. GetRecordAsync returns null on a 404 and DeleteRecordAsync returns false; every other non-success status is an InferHubException. Needs the coordinator running with VectorStore:Enabled=true and the collection already created.

await client.UpsertAsync("docs", VectorUpsert
    .FromText("doc-1", "InferHub is a self-hosted inference mesh.", "nomic-embed-text"));

var matches = await client.QueryAsync("docs",
    VectorQuery.FromText("what is InferHub?", "nomic-embed-text", k: 3));

// matches[0].Score, matches[0].Id - ranked, closest first.

RAG retrieval (v0.5.0). Chat and generate can now be grounded in a collection per call. Pass an InferHubCallOptions — it maps to the coordinator's X-InferHub-Retrieve[-K|-Model] headers, so the request body stays untouched — and the coordinator pulls the top matches, grounds the prompt, and echoes the record ids it used in X-InferHub-Sources. The client parses those into SourceIds on the response (works for blocking and streaming). It ships as overloads, so a call with no options behaves exactly like a plain chat. A retrieval that cannot be satisfied — store unavailable, or OnMissing=error with no hits — fails with HTTP 424 as an InferHubRetrievalException (a subclass of InferHubException), never a silent, ungrounded answer. A per-call ConversationId maps to X-InferHub-Conversation for sticky routing.

var response = await client.ChatAsync(
    new ChatRequest
    {
        Model = "llama3",
        Messages = new[] { new ChatMessage { Role = "user", Content = "How do nodes connect?" } }
    },
    InferHubCallOptions.ForRetrieval("docs", k: 3, model: "nomic-embed-text"));

Console.WriteLine(response.Message?.Content);
Console.WriteLine(string.Join(", ", response.SourceIds ?? [])); // grounding record ids

What v0.5.0 covers. Blocking + streaming ChatAsync/ChatStreamAsync and GenerateAsync/GenerateStreamAsync, EmbedAsync (batch) and EmbedLegacyAsync, the vector data-plane (UpsertAsync, QueryAsync, RetrieveAsync, GetRecordAsync, DeleteRecordAsync), opt-in RAG retrieval on chat/generate (InferHubCallOptions → source ids on the response), ListModelsAsync (/api/tags), GetStatusAsync, and a cheap PingAsync against /health. Request models carry a [JsonExtensionData] bag, so anything the Ollama contract accepts — options, tool definitions, format, keep_alive — passes through untouched. Non-success responses surface as InferHubException with the coordinator's { "error": … } body and the HTTP status.

What is next. A separate admin client for fleet control (cordon/drain, deregister) and collection lifecycle (create/drop/rebuild), plus a live SSE event stream — kept on its own interface so a client key never surfaces admin methods. The public surface added in v0.1.0 stays stable as later phases layer on top.

Roadmap & status

InferHub 2.0 closes the RAG mesh: models can read from the vector store inline. Add X-InferHub-Retrieve: <collection> to a normal /api/chat or /api/generate call and the coordinator embeds the query on a GPU node, searches the store (node replica if available, hub-local otherwise), assembles a grounded prompt, and dispatches the generation to a node. Retrieval is opt-in per request — without the header, calls are byte-for-byte identical to v1.x. This sits on top of v1.9's console surface, v1.8's self-healing replicas, v1.7's replication push-protocol, v1.6's text-in/vectors-out loop, the v1.5 vector store, and the v1.4 base (Bearer auth, sticky + least-busy routing, live streaming, pre-stream failover, typed/validated node config, an admin API, and a live management console).
